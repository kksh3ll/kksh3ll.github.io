> 当代码不再能被业务读懂，DDD让代码重新说业务的语言

## 一、DDD想解决的核心问题

### 1. 业务语言与代码语言割裂

产品经理说"待支付订单"，开发人员翻开代码看到的却是 `status == 1`。一个新入职的同事问"订单发货后还能不能取消"，得到的回答是"你得去 Controller 里看一遍校验逻辑，再去 Service 里看一遍流程，哦对了，那个定时任务里好像也有一份"。新人接手全靠口口相传，系统变成了只有"老人"才能读懂的黑盒。

### 2. 业务规则散落

一个规则如"订单不能在发货后取消"，可能同时出现在 Controller 校验、Service 流程中，甚至定时任务里。改一处漏三处成了家常便饭，线上事故往往就是这么来的。

```java
// Controller里的校验
if (order.getStatus() == OrderStatus.SHIPPED) {
    throw new BizException("已发货订单不可取消");
}

// Service流程里的校验
if (order.getStatus() == OrderStatus.SHIPPED) {
    throw new BizException("已发货订单不可取消");
}

// 定时任务里的校验
if (order.getStatus() == OrderStatus.SHIPPED) {
    log.warn("已发货订单不可取消, orderId={}", order.getId());
    continue;
}
```

同一个业务规则出现了三次，谁敢保证第四次需求变更时不会漏改？

### 3. 边界由技术决定而不是由业务决定

模块按"用户管理/商品管理"等表名拆分，却和真实业务流程（下单、结算）对不齐。跨表一致性事务增多，一个小小的下单操作可能要在一个事务里同时操作订单表、库存表、积分表、优惠券表。

## 二、从三层到四层：一次认识论的升级

传统三层架构——Controller/Service/Dao——以技术视角分层：接口管协议，业务管流程，数据管存取。Model 退化成只有 getter/setter 的数据容器，"下单"变成了 Service 里一段过程式的流程编排，而不是 Order 对象自身的行为。

DDD 将其演进为四层：

| 层 | 传统架构 | DDD架构 |
|---|---|---|
| 接口层 | Controller、API | Controller、ViewObject |
| 应用层 | DTO、Service | DTO、Application Service |
| 领域层 | Model、Dao | **DomainService、ValueObject、Entity、Repository（接口）** |
| 基础设施层 | — | **RepositoryImpl、EventMessage** |

关键变化在于：**领域层被独立出来**，它不再被持久化技术绑架，而是专注于业务规则本身；基础设施层从"被业务依赖"变成"实现领域定义的接口"，依赖方向翻转。

## 三、三大思想内核

### 1. 统一语言：让所有人说同一套话

领域专家、产品经理和研发在同一个限界上下文中使用同一套术语体系，并且这套术语直接映射到文档、类名和方法名中。

"待支付订单"不再对应 `status == 1`，而是 `OrderStatus.PENDING_PAYMENT`；"取消已发货订单"不再是散落各处的 `if` 校验，而是 `order.cancel()` 在聚合根内部完成校验和状态迁移。**代码即文档，文档即业务**。

### 2. 领域模型：从贫血到充血

领域模型是业务规则在代码里的精确映射——封装了数据、行为和业务不变量的对象。

贫血模型的本质是过程式设计的产物：Order 只持有数据，所有行为散落在 OrderService 中。充血模型则体现了面向对象的本质：**对象应基于自身状态执行业务动作，并保护自己的不变量**。

```java
// ========== 贫血模型：行为外置，不变量无人守护 ==========
public class Order {
    private Integer status;
    private BigDecimal totalAmount;
    // 只有getter/setter，没有行为
}

public class OrderService {
    public void pay(Long orderId, String paymentId) {
        Order order = orderDao.findById(orderId);
        // 校验散落在Service中，任何调用者都可能绕过
        if (order.getStatus() != OrderStatus.PENDING_PAYMENT.getCode()) {
            throw new BizException("订单状态不正确");
        }
        if (order.getTotalAmount().compareTo(BigDecimal.ZERO) <= 0) {
            throw new BizException("订单金额异常");
        }
        order.setStatus(OrderStatus.PAID.getCode());
        orderDao.update(order);
    }
}

// 任何地方都可以绕过校验直接改状态
order.setStatus(OrderStatus.PAID.getCode()); // 不变量被随意破坏
```

```java
// ========== 充血模型：行为内聚，不变量被封装保护 ==========
public class Order {
    private OrderStatus status;
    private Money totalAmount;

    // 状态迁移的入口唯一、规则不可绕过
    public void pay(PaymentId paymentId) {
        if (this.status != OrderStatus.PENDING_PAYMENT) {
            throw new DomainException("只有待支付订单才能支付");
        }
        if (this.totalAmount.isNegativeOrZero()) {
            throw new DomainException("订单金额异常");
        }
        this.status = OrderStatus.PAID;
        this.paymentId = paymentId;
        DomainEventPublisher.publish(new OrderPaidEvent(this.id, paymentId));
    }
}
```

区别不在于代码行数，而在于**谁对业务规则负责**。贫血模型下，任何调用者都可以随意 `setStatus`，不变量的保护完全依赖调用方的自觉；充血模型下，状态迁移的入口唯一、规则不可绕过。

### 3. 限界上下文：允许同名异义

划定业务边界，在边界内保持模型的简洁与一致，**允许同一个词在不同上下文中拥有不同含义**。

以"商品"为例：

| 限界上下文 | "商品"的含义 |
|---|---|
| 销售上下文 | 价格、促销标签、库存数 |
| 物流上下文 | 体积、重量、包装规格 |

两个上下文各自建模、各自演进，不必为了"统一商品模型"而制造一个包含所有字段的大杂烩。限界上下文让模型在边界内自由呼吸，在边界间明确协作。

## 四、战略设计：划疆而治

### 1. 领域、子域与核心域

领域是组织关注的业务范围，子域是领域内的独立业务板块。DDD 将子域分为三类：

| 类型 | 定位 | 举例 |
|---|---|---|
| 核心域 | 核心竞争力，投入最多资源 | 订单、交易 |
| 支撑域 | 业务必须但非差异化 | 商品、评论 |
| 通用域 | 可被多个子域共享 | 权限、登录、消息推送 |

**核心域决定你能不能赢，支撑域决定你能不能跑，通用域决定你是不是在重复造轮子。** 核心域值得最优秀的人、最精细的建模；通用域能买就不建；支撑域保持够用即可。

### 2. 限界上下文与上下文映射

识别子域后，还需回答两个问题：哪几个子域合在一个限界上下文里？上下文之间用什么模式协作？

常见的协作模式：

- **共享内核（Shared Kernel）**：两个上下文共享一小段代码或模型，代价是共享部分变更需双方协商
- **客户-供应商（Customer-Supplier）**：下游依赖上游，但下游可影响上游的演进方向
- **追随者（Conformist）**：下游完全适配上游模型，放弃自主权换取简单性
- **防腐层（Anticorruption Layer）**：在边界处加翻译层，隔离外部模型对内部模型的污染——这是最值得投入的协作模式之一
- **开放主机服务（Open Host Service）**：上游提供标准协议供多方接入
- **发布语言（Published Language）**：用公开标准格式（如 JSON Schema、Protocol Buffers）作为通信契约

防腐层的典型实现：

```java
// 外部支付系统的模型——与我们内部的领域模型完全不同
public class ExternalPaymentResponse {
    private String respCode;     // "0000"表示成功
    private String tradeNo;      // 外部交易号
    private BigDecimal amt;      // 金额
}

// 防腐层：将外部模型翻译为内部领域模型
public class PaymentGatewayAdapter implements PaymentGateway {
    public PaymentResult pay(PaymentCommand command) {
        ExternalPaymentRequest request = translateToExternal(command);
        ExternalPaymentResponse response = externalPaymentClient.pay(request);
        // 翻译：外部模型 -> 内部模型，隔离污染
        return translateToDomain(response);
    }

    private PaymentResult translateToDomain(ExternalPaymentResponse resp) {
        if ("0000".equals(resp.getRespCode())) {
            return PaymentResult.success(new PaymentId(resp.getTradeNo()),
                                          new Money(resp.getAmt()));
        }
        return PaymentResult.fail(resp.getRespCode());
    }
}
```

上下文映射不是架构图上的装饰，而是团队协作契约的可视化。它回答了"谁依赖谁、依赖什么、如何隔离"这三个最易引发混乱的问题。

## 五、战术设计：边界内的对象组织

战略设计划定了疆界，战术设计回答的是：在一个限界上下文内，对象如何组织、业务规则放在哪里。

### 1. 实体（Entity）

有唯一标识、有生命周期、可变。判断方法——如果两个对象除了 ID 完全一样，业务上仍然是两个东西，那就是实体。

```java
public class Order {
    private OrderId id;           // 唯一标识
    private OrderStatus status;   // 可变状态
    private List<OrderItem> items;

    // 即使两个Order的所有属性都一样，只要id不同就是不同的Order
}
```

### 2. 值对象（Value Object）

无唯一标识、不可变、按属性值判等。典型例子：金额、地址、日期范围。

值对象的重要性常被低估。它把裸加减乘除散落各处的代码变成 `Money` 自己负责货币规则——"金额不能为负"、"不同币种不能直接相加"这些规则从散落的 `if` 判断变成了值对象的内置行为。

```java
// 没有值对象时：金额逻辑散落各处
public class OrderService {
    public void refund(Long orderId) {
        Order order = orderDao.findById(orderId);
        BigDecimal refundAmount = order.getAmount();
        if (refundAmount.compareTo(BigDecimal.ZERO) <= 0) {
            throw new BizException("退款金额必须大于0");
        }
        // 还要考虑币种？精度？四舍五入规则？
        accountService.addBalance(order.getUserId(), refundAmount);
    }
}
```

```java
// 有值对象时：金额规则内聚到Money中
public class Money {
    private final BigDecimal amount;
    private final Currency currency;

    public Money(BigDecimal amount, Currency currency) {
        if (amount == null) throw new DomainException("金额不能为null");
        if (currency == null) throw new DomainException("币种不能为null");
        this.amount = amount.setScale(currency.getDefaultFractionDigits(), RoundingMode.HALF_UP);
        this.currency = currency;
    }

    public Money add(Money other) {
        assertSameCurrency(other);
        return new Money(this.amount.add(other.amount), this.currency);
    }

    public Money negate() {
        return new Money(this.amount.negate(), this.currency);
    }

    public boolean isNegativeOrZero() {
        return this.amount.compareTo(BigDecimal.ZERO) <= 0;
    }

    private void assertSameCurrency(Money other) {
        if (!this.currency.equals(other.currency)) {
            throw new DomainException("不同币种不能直接运算: " + this.currency + " vs " + other.currency);
        }
    }
}

// 使用：金额规则不再散落
public void refund(OrderId orderId) {
    Order order = repository.findById(orderId);
    Money refundAmount = order.getTotalAmount();
    accountService.addBalance(order.getUserId(), refundAmount);
}
```

**这是减少魔法字符串、魔数和重复校验的有效手段。**

### 3. 聚合与聚合根（Aggregate & Aggregate Root）

**聚合**是一组业务上不可分割、共享生命周期的实体与值对象的集合。**聚合根**是聚合的唯一对外入口，负责维护聚合内部的不变量。

```java
// Order聚合：Order是聚合根，OrderItem是内部实体，Money/Address是值对象
public class Order extends AggregateRoot<OrderId> {
    private OrderId id;
    private OrderStatus status;
    private List<OrderItem> items;       // 内部实体，外部不应直接操作
    private Money totalAmount;           // 值对象
    private ShippingAddress address;     // 值对象

    // 聚合根维护不变量：订单总金额必须等于明细之和
    public void addItem(Product product, int quantity) {
        if (this.status != OrderStatus.PENDING_PAYMENT) {
            throw new DomainException("只有待支付订单才能添加商品");
        }
        OrderItem item = new OrderItem(product.getId(), product.getPrice(), quantity);
        this.items.add(item);
        this.totalAmount = recalculateTotal();  // 不变量：总额与明细一致
    }

    public void cancel(String reason) {
        if (this.status == OrderStatus.SHIPPED) {
            throw new DomainException("已发货订单不可取消");
        }
        if (this.status == OrderStatus.CANCELLED) {
            throw new DomainException("订单已取消");
        }
        this.status = OrderStatus.CANCELLED;
        this.cancelReason = reason;
        registerEvent(new OrderCancelledEvent(this.id, reason));
    }

    private Money recalculateTotal() {
        return items.stream()
            .map(OrderItem::getSubtotal)
            .reduce(Money.ZERO(this.totalAmount.getCurrency()), Money::add);
    }
}
```

三条设计原则：

1. **聚合外部尽量只持有聚合根的 ID 引用**，避免直接持有内部实体引用——防止外部绕过聚合根修改内部状态

```java
// 反模式：外部直接持有内部实体引用
public class Shipment {
    private OrderItem item;  // 持有了Order的内部实体！
}

// 正确做法：只持有聚合根ID
public class Shipment {
    private OrderId orderId;  // 只持有聚合根的ID引用
}
```

2. **一个事务优先只修改一个聚合**——跨聚合一致性通过领域事件和最终一致性处理，而非优先诉诸分布式事务

3. **聚合要小**——大聚合带来并发竞争、加载成本和性能问题

判断聚合大小的经验法则：**如果一个不变量需要在一次事务内保持强一致，它涉及的对象通常应放在一个聚合内；否则优先拆开。**

判断一致性边界时的三个试金石：

1. **规则是否立刻成立？** 不立刻成立就会造成业务错误的，倾向放入同一聚合
2. **规则是否只依赖内部数据？** 需要查询大量外部对象的，通常说明边界过大
3. **变化是否需要同一事务提交？** 可以通过事件最终一致的，不必把多个聚合塞到一起

常见反例：把订单、支付、库存、物流、积分全塞到一个 `Order` 聚合。结果就是锁竞争、对象加载过重、团队边界混乱。**更推荐用小聚合保护核心不变量，用领域事件连接跨聚合流程。**

### 4. 领域服务（Domain Service）

当一个领域行为不自然地属于任何单个实体或值对象时，放到领域服务里。典型场景：

- 涉及多个聚合的业务规则（如订单转移：涉及 Order 与 User）
- 需要查询仓储才能完成的领域判断（如用户的待支付订单是否超限）

```java
// 涉及多个聚合的领域规则：订单转移
public class OrderTransferService {
    private final OrderRepository orderRepository;
    private final UserRepository userRepository;

    public void transfer(OrderId orderId, UserId fromUser, UserId toUser) {
        Order order = orderRepository.findById(orderId);
        User source = userRepository.findById(fromUser);
        User target = userRepository.findById(toUser);

        // 跨聚合的业务规则：不属于任何单个实体
        if (!order.getOwnerId().equals(fromUser)) {
            throw new DomainException("只能转移自己的订单");
        }
        if (source.getTransferCount() >= MAX_TRANSFER_PER_DAY) {
            throw new DomainException("已达每日转移上限");
        }
        if (target.isBlacklisted()) {
            throw new DomainException("目标用户已被拉黑");
        }

        order.transferTo(toUser);
        source.incrementTransferCount();
        target.receiveOrder(orderId);
    }
}
```

```java
// 需要查询仓储的领域判断
public class OrderLimitService {
    private final OrderRepository orderRepository;

    public void checkPendingPaymentLimit(UserId userId) {
        int pendingCount = orderRepository.countByUserAndStatus(userId, OrderStatus.PENDING_PAYMENT);
        if (pendingCount >= MAX_PENDING_ORDERS) {
            throw new DomainException("待支付订单数已达上限");
        }
    }
}
```

**反模式警告**：不要把本该在实体上的行为（如 `Order.pay()`）挪到领域服务里，否则实体会退化成贫血模型。**优先放实体或值对象，确定不属于任何对象时再放领域服务。**

```java
// 反模式：本该在实体上的行为被挪到了领域服务
public class OrderDomainService {
    public void pay(Order order, PaymentId paymentId) {
        if (order.getStatus() != OrderStatus.PENDING_PAYMENT) {
            throw new DomainException("只有待支付订单才能支付");
        }
        order.setStatus(OrderStatus.PAID);  // Order退化成了数据容器！
        order.setPaymentId(paymentId);
    }
}

// 正确做法：行为在实体上
public class Order {
    public void pay(PaymentId paymentId) {
        if (this.status != OrderStatus.PENDING_PAYMENT) {
            throw new DomainException("只有待支付订单才能支付");
        }
        this.status = OrderStatus.PAID;
        this.paymentId = paymentId;
        registerEvent(new OrderPaidEvent(this.id, paymentId));
    }
}
```

### 5. 领域事件（Domain Event）

领域事件是领域内已经发生的、值得让其他部分知道的事实。它带来三重价值：

**解耦**：订单不需要知道"支付完成后要发短信、扣库存、加积分"，它只发布 `OrderPaidEvent`，下游各自订阅。

```java
// 聚合根内发布领域事件
public class Order extends AggregateRoot<OrderId> {
    public void pay(PaymentId paymentId) {
        // ... 校验和状态变更 ...
        this.status = OrderStatus.PAID;
        this.paymentId = paymentId;
        registerEvent(new OrderPaidEvent(this.id, paymentId, this.totalAmount));
    }
}

// 应用服务层发布事件
public class OrderApplicationService {
    @Transactional
    public void pay(PayOrderCommand cmd) {
        Order order = orderRepository.findById(cmd.getOrderId());
        order.pay(cmd.getPaymentId());
        orderRepository.save(order);
        domainEventPublisher.publishAll(order.getDomainEvents());
    }
}

// 各下游各自订阅，互不依赖
public class InventoryEventHandler {
    @EventHandler
    public void on(OrderPaidEvent event) {
        inventoryService.deductStock(event.getOrderId(), event.getItems());
    }
}

public class PointsEventHandler {
    @EventHandler
    public void on(OrderPaidEvent event) {
        pointsService.earnPoints(event.getUserId(), event.getTotalAmount());
    }
}

public class NotificationEventHandler {
    @EventHandler
    public void on(OrderPaidEvent event) {
        notificationService.sendPaymentSuccess(event.getUserId(), event.getOrderId());
    }
}
```

**可追溯**：领域事件记录了业务事实，在合适的架构中可以进一步发展为事件溯源（Event Sourcing）。

**异步友好**：事件天然适配消息队列，是构建最终一致性的基石。

### 6. 仓储（Repository）与工厂（Factory）

**仓储**提供像访问内存集合一样的持久化抽象。关键约束：**仓储接口定义在领域层，实现在基础设施层**——这是依赖反转原则的直接体现。

```java
// 领域层：定义仓储接口
public interface OrderRepository {
    Order findById(OrderId id);
    void save(Order order);
    void remove(Order order);
    List<Order> findByUserId(UserId userId);
}

// 基础设施层：实现仓储
public class JpaOrderRepository implements OrderRepository {
    private final JpaOrderDao jpaOrderDao;

    @Override
    public Order findById(OrderId id) {
        OrderPO po = jpaOrderDao.findById(id.getValue()).orElse(null);
        return OrderConverter.toDomain(po);  // 持久化对象 -> 领域对象
    }

    @Override
    public void save(Order order) {
        OrderPO po = OrderConverter.toPO(order);  // 领域对象 -> 持久化对象
        jpaOrderDao.save(po);
    }
}
```

**工厂**处理聚合的复杂创建逻辑。当创建涉及多个对象的初始化和业务校验时，将其抽取到工厂中，避免聚合根构造函数膨胀。

```java
public class OrderFactory {
    public static Order createOrder(UserId userId, List<OrderItemCommand> itemCommands,
                                     ShippingAddress address, CouponId couponId) {
        // 复杂的创建逻辑：校验、初始化、计算
        if (itemCommands.isEmpty()) {
            throw new DomainException("订单不能为空");
        }
        List<OrderItem> items = itemCommands.stream()
            .map(cmd -> new OrderItem(cmd.getProductId(), cmd.getPrice(), cmd.getQuantity()))
            .collect(Collectors.toList());

        Money totalAmount = items.stream()
            .map(OrderItem::getSubtotal)
            .reduce(Money.ZERO(items.get(0).getCurrency()), Money::add);

        if (couponId != null) {
            totalAmount = applyCoupon(totalAmount, couponId);
        }

        Order order = new Order(OrderId.generate(), userId, items, totalAmount, address);
        order.registerEvent(new OrderCreatedEvent(order.getId(), userId, totalAmount));
        return order;
    }
}
```

## 六、分层架构：领域居中，依赖向内

### 1. 四层职责与边界

| 层 | 应该做 | 不应该做 |
|---|---|---|
| 用户界面层 | 协议解析、入参出参格式、调用应用服务 | 写业务规则、绕过应用服务 |
| 应用服务层 | 用例编排、事务边界、DTO 转换、发布事件 | 承载核心业务规则、写 SQL |
| 领域层 | 业务规则、不变量、状态迁移、领域事件 | 引用持久化技术、HTTP 客户端、MQ SDK |
| 基础设施层 | 仓储实现、ORM、消息队列适配器、外部 API 客户端 | 包含业务规则、定义业务概念 |

### 2. 最关键的设计原则：领域层不应该依赖基础设施层

```java
// 领域层定义接口
public interface OrderRepository {
    Order findById(OrderId id);
    void save(Order order);
}

// 基础设施层实现
public class JpaOrderRepository implements OrderRepository { ... }

// 应用服务层注入接口、调用方法
public class OrderApplicationService {
    private final OrderRepository orderRepository;  // 依赖接口，不依赖实现

    public OrderApplicationService(OrderRepository orderRepository) {
        this.orderRepository = orderRepository;
    }
}
```

依赖方向：用户界面层 → 应用服务层 → 领域层 ← 基础设施层。**基础设施层依赖领域层，而非反过来**。

### 3. 分层架构的演化：领域居中，依赖向内

**六边形架构（端口-适配器）**：所有外部世界（HTTP、数据库、消息队列、第三方 API）都作为适配器，通过端口接入核心。驱动端口通常是应用服务接口，被驱动端口通常是仓储、消息发布等。左侧适配器将外部协议（HTTP、gRPC、CLI）翻译为驱动端口调用，右侧适配器将被驱动端口调用翻译为外部协议（SQL、Kafka、REST）。

**洋葱架构**：最内圈是领域模型，向外依次是领域服务、应用服务、基础设施。所有依赖指向内层，内圈不允许引用外圈。

**整洁架构**：源代码依赖只能指向内层。最内圈 Entities（领域模型），第二圈 Use Cases（应用服务），第三圈 Controller/Presenter/Gateway，最外层框架与驱动。与洋葱架构高度相似，但更关注普适的应用架构。

**显式架构**：面向现代工程实践的综合视图。应用核心 = 洋葱内三圈 = 整洁内两圈 = 六边形核心区域；入口和出口 = 六边形的左右两类适配器；CQRS/领域事件/外部事件分发 = DDD 战术模式。

四条路径，一个共识：**领域模型居于核心；依赖方向始终向内；通过适配器实现解耦。**

### 4. CQRS 和 Event Sourcing：复杂读写与历史溯源

当系统的读写复杂度差异显著时，CQRS（Command Query Responsibility Segregation）将写模型和读模型分开：

- **写入侧**关心业务规则、不变量、状态迁移，适合围绕聚合建模
- **查询侧**关心页面展示、列表筛选、统计报表、跨表聚合

```java
// ========== 写入侧：围绕聚合建模，走领域模型 ==========
public class OrderApplicationService {
    @Transactional
    public void createOrder(CreateOrderCommand cmd) {
        Order order = OrderFactory.create(cmd);
        orderRepository.save(order);
        domainEventPublisher.publishAll(order.getDomainEvents());
    }
}

// ========== 查询侧：直接面向读模型，不必受聚合边界约束 ==========
public class OrderQueryService {
    private final JdbcTemplate jdbcTemplate;

    public OrderListVO queryOrders(OrderQuery query) {
        // 查询侧可以直接写SQL做跨表聚合、分页、排序
        String sql = "SELECT o.id, o.status, o.total_amount, u.name as user_name, " +
                     "p.name as product_name FROM t_order o " +
                     "LEFT JOIN t_user u ON o.user_id = u.id " +
                     "LEFT JOIN t_order_item oi ON o.id = oi.order_id " +
                     "LEFT JOIN t_product p ON oi.product_id = p.id " +
                     "WHERE o.user_id = ? ORDER BY o.created_at DESC LIMIT ?, ?";
        return jdbcTemplate.query(sql, new OrderListVORowMapper(),
                                  query.getUserId(), query.getOffset(), query.getLimit());
    }
}
```

Event Sourcing 则更进一步：**不存储当前状态，而是存储所有改变状态的事件**。状态的获取通过回放事件实现。

```java
// 传统方式：只存当前状态
// t_order: id=1, status=PAID, amount=99.00

// Event Sourcing：存储状态变化历史
// t_order_event:
//   id=1, order_id=1, type=ORDER_CREATED,   data={items:[...], amount:99.00}, timestamp=...
//   id=2, order_id=1, type=ORDER_PAID,      data={paymentId:P001},            timestamp=...
//   id=3, order_id=1, type=ORDER_SHIPPED,   data={trackingNo:SF123},          timestamp=...

public class EventSourcedOrder {
    private OrderId id;
    private OrderStatus status;
    private Money totalAmount;
    private List<DomainEvent> changes = new ArrayList<>();

    // 从事件流重建状态
    public static EventSourcedOrder fromEvents(List<DomainEvent> events) {
        EventSourcedOrder order = new EventSourcedOrder();
        events.forEach(order::apply);
        order.changes.clear();  // 重建时的事件不算新变更
        return order;
    }

    // 每个事件触发状态迁移
    private void apply(DomainEvent event) {
        if (event instanceof OrderCreatedEvent e) {
            this.id = e.getOrderId();
            this.status = OrderStatus.PENDING_PAYMENT;
            this.totalAmount = e.getTotalAmount();
        } else if (event instanceof OrderPaidEvent e) {
            this.status = OrderStatus.PAID;
        } else if (event instanceof OrderShippedEvent e) {
            this.status = OrderStatus.SHIPPED;
        }
        // 回溯任意时刻的状态：只需取截止该时刻的事件回放
    }
}
```

它保护的是状态变化的历史——每一笔"为什么变成这样"都有据可查，天然支持审计、回溯和时间旅行调试。

CQRS 与 Event Sourcing 常被一起使用，但它们是正交的：可以只用 CQRS 不做事件溯源，也可以做事件溯源但不分离读写模型。选择取决于业务复杂度，而非架构审美。

## 写在最后

DDD 不是框架，不是方法论，而是一种**让代码重新说业务语言**的思维方式。

统一语言消除沟通损耗，限界上下文划定协作边界，领域模型让业务规则回到它该在的地方，分层架构保护领域核心不被技术细节侵蚀。战略设计解决"画地图"的问题，战术设计解决"盖房子"的问题，而贯穿始终的，是对业务本质的尊重。

从 `status == 1` 到 `OrderStatus.PENDING_PAYMENT`，从散落的 `if` 校验到聚合根内的不变量守护，从跨表大事务到小聚合加领域事件的最终一致性——每一步，都是让代码离业务更近一步。

**代码应该像业务文档一样可读，业务规则应该像领域模型一样不可绕过。** 这是 DDD 给出的承诺，也是它值得投入的理由。
