## 现有方案调研

在决定自研之前，我们系统性地调研了主流的开源告警方案，评估它们与我们需求的匹配程度。

### 1. Prometheus Alerting Rules

Prometheus 自带的告警规则是最常见的方案。它在 `alerting_rules.yml` 中定义规则，Prometheus 引擎周期性评估，触发后推送到 Alertmanager。

**优势**：
- 与 Prometheus 生态深度集成
- 完整的 PromQL 支持
- `for` 子句支持持续时长判断
- Alertmanager 提供分组、抑制、静默等高级功能

**不足**：
- 规则通过配置文件管理，无法运行时动态增删
- 不原生支持多租户规则隔离
- 通知渠道依赖 Alertmanager，自定义渠道需要开发 AM 的 receiver 插件
- 无法直接对接 VictoriaMetrics 的多租户查询 API
- 规则变更需要 reload，在大量规则场景下 reload 周期不可控

### 2. VictoriaMetrics vmalert

vmalert 是 VictoriaMetrics 官方的告警组件，功能定位与 Prometheus alerting rules 类似，但直接对接 VM 数据源。

**优势**：
- 原生支持 VictoriaMetrics 数据源
- 支持多租户（通过 tenant HTTP header）
- 兼容 Prometheus 告警规则语法
- 支持 recording rules

**不足**：
- 同样依赖配置文件，动态性不足
- 通知依赖 Alertmanager 或 VictoriaMetrics alertmanager
- 无法按规则粒度配置不同通知渠道
- 规则表达式严格遵循 PromQL，无法做业务层面的逻辑组合扩展
- 在我们的场景下，规则的动态管理需求远超 vmalert 的能力边界

### 3. Grafana Alerting

Grafana 从 v8 开始重构了告警系统，提供了统一的告警管理界面和 API。

**优势**：
- UI 友好，可视化规则编辑
- 支持多数据源
- 内置通知模板和多通道支持

**不足**：
- 架构偏重，引入 Grafana 作为告警组件对运维成本有显著增加
- 规则评估依赖 Grafana 内部调度器，性能和可控性不如独立引擎
- 多租户能力受限于 Grafana Organization 体系
- 定制化困难，Grafana 的告警 API 不够灵活

### 4. 开源规则引擎（如 Cortexpr/Thanos Ruler）

Thanos Ruler 和 Cortex Ruler 提供了分布式告警评估能力，但它们的核心目标是将 Prometheus 的规则评估水平扩展，而非提供更灵活的规则管理体验。

**结论**：这些方案解决的是"规模"问题，而我们面临的是"灵活性"问题。

## 需求矩阵

将我们的核心需求与各方案对照：

| 需求 | Prometheus | vmalert | Grafana | 自研 |
|------|-----------|---------|---------|------|
| API 动态规则管理 | ❌ | ❌ | ✅ | ✅ |
| VictoriaMetrics 对接 | ⚠️ 需适配 | ✅ | ✅ | ✅ |
| 多租户规则隔离 | ❌ | ✅ | ⚠️ | ✅ |
| 按规则配置通知渠道 | ❌ | ❌ | ✅ | ✅ |
| 自定义表达式扩展 | ❌ | ❌ | ❌ | ✅ |
| 轻量部署 | ✅ | ✅ | ❌ | ✅ |
| 持续时长（for）支持 | ✅ | ✅ | ✅ | ✅ |
| 告警状态生命周期 | ✅ | ✅ | ✅ | ✅ |

没有方案能同时满足所有需求，尤其是在 **API 动态规则管理** 和 **自定义表达式扩展** 这两个维度上，所有现有方案都存在根本性缺失。

## 核心架构决策

### 决策一：自建表达式引擎 vs 嵌入 PromQL 引擎

**选项 A**：直接复用 Prometheus 的 PromQL 引擎代码（`prometheus/prometheus/promql`）

- 优势：完整的 PromQL 支持，经过大规模生产验证
- 劣势：引入 Prometheus 整个依赖树；PromQL 引擎与 Prometheus 存储耦合深，抽取困难；无法灵活扩展业务函数

**选项 B**：使用第三方 PromQL 解析库（如 `promql-parser`）

- 优势：轻量解析
- 劣势：只有解析能力，无求值能力；求值仍需对接 VM API

**选项 C**：自建表达式引擎，解析类 PromQL 语法，求值时调用 VM 查询 API

- 优势：完全可控，可按需扩展函数和语法；依赖极简；查询结果可直接映射为内部 Sample 结构
- 劣势：需要自行实现词法分析、语法解析、AST 求值；PromQL 兼容性有限

**最终选择：选项 C**。理由如下：

1. 我们的场景不需要 100% PromQL 兼容——只需要覆盖选择器、聚合、比较、逻辑运算和常用函数。
2. 自建引擎使得我们可以自由扩展业务函数（如预测性告警函数），而无需 fork PromQL。
3. 求值逻辑通过 VM 的即时查询 API（`/query`）和范围查询 API（`/query_range`）完成，我们只需要解析表达式结构、编排查询、组合结果。

### 决策二：规则存储方案

**选项 A**：数据库（PostgreSQL/MySQL）

- 优势：持久化、事务支持、成熟
- 劣势：引入外部依赖，与"轻量部署"目标冲突

**选项 B**：嵌入式数据库（SQLite/BoltDB）

- 优势：无需外部依赖，单文件存储
- 劣势：并发写入性能有限，多实例部署时数据不共享

**选项 C**：内存存储 + 接口抽象

- 优势：最快实现，通过 Store 接口抽象，后续可无缝切换到任何持久化方案
- 劣势：进程重启后规则丢失

**最终选择：选项 C**。我们定义了 `Store` 接口，先用 `MemoryStore` 实现核心功能，持久化作为后续迭代目标。这遵循了"先跑通核心链路，再补齐持久化"的渐进式开发策略。

### 决策三：调度模型

**选项 A**：全局定时轮询

- 固定间隔遍历所有规则，逐条评估
- 简单但无法支持不同规则的不同评估间隔

**选项 B**：每规则独立 goroutine + ticker

- 每条规则启动独立的 goroutine，按规则自身的 interval 周期性评估
- 支持不同规则不同间隔，但 goroutine 数量与规则数量成正比

**选项 C**：时间轮/优先队列调度

- 精确调度，低开销
- 实现复杂度高

**最终选择：选项 B**。我们的规则规模在数百级别，goroutine 开销完全可接受。每规则独立 goroutine 的模型天然支持不同的评估间隔，且��则增删时启停 goroutine 的实现简单直观。

### 决策四：告警通知模型

**选项 A**：推模型（评估后立即通知）

- 评估完成即推送，延迟最低
- 可能产生告警风暴

**选项 B**：推模型 + 去重/静默

- 在推送前增加去重和静默逻辑
- 需要维护额外的去重状态

**选项 C**：推模型 + 状态对比（新 firing / 新 resolved）

- 每次评估后与上一次状态对比，仅对新出现的 firing 或 resolved 发送通知
- 天然去重，逻辑简洁

**最终选择：选项 C**。在 `handleAlerts` 方法中，我们将本次评估产生的告警集合与上次告警集合做 diff——新出现的告警标记为 firing 并通知，消失的告警标记为 resolved 并通知。这个模型简单且有效。

## 技术风险评估

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 自建表达式引擎的 PromQL 兼容性不足 | 用户迁移成本高 | 覆盖最常用的 80% 语法，未覆盖的场景提示明确错误 |
| 内存存储规则丢失 | 重启后需重新创建规则 | 提供批量导入 API；后续迭代增加持久化 |
| 大量规则时 goroutine 开销 | 内存和调度压力 | 压测验证；必要时切换为 worker pool 模型 |
| Webhook 重试可能产生重复通知 | 接收方需幂等处理 | 通知携带 alert ID 和时间戳，接收方可据此去重 |

## 总结

技术选型的核心思路是 **渐进式构建**——先用最简方案验证核心链路（表达式引擎 → 调度器 → 通知分发），再逐步补齐持久化、高可用、可观测性等生产级能力。这种策略让我们在第一版就能快速验证端到端流程，同时通过接口抽象为后续演进保留了充足的空间。
