# 项目起源与背景

## 监控告警的困境

在现代云原生运维体系中，监控和告警是保障服务稳定性的核心环节。我们的运维团队管理着数十个微服务，部署在 Kubernetes 集群上，使用 VictoriaMetrics 作为时序数据存储。随着业务规模扩张，我们面临了越来越严峻的告警管理问题。

起初，我们像许多团队一样，在 Prometheus/VictoriaMetrics 的配置文件中硬编码告警规则。这种方式在规则数量较少时可以勉强工作，但随着业务增长，问题逐一暴露：

- **规则管理混乱**：数百条告警规则散落在多个 YAML 文件中，修改一条规则需要提交 PR、等待合并、触发 reload，整个流程耗时且容易出错。
- **缺乏动态性**：无法在运行时增删改规则，每次变更都需要重启或发送 SIGHUP 信号。
- **多租户需求缺失**：不同业务线需要独立的告警规则管理能力，但配置文件方案无法做到租户隔离。
- **告警通知渠道单一**：只支持 Alertmanager 的固定通知路径，无法灵活地为不同规则配置 Webhook、邮件、钉钉等不同通知渠道。
- **表达式能力受限**：VictoriaMetrics 自带的告警功能虽然支持 PromQL，但缺乏对复杂逻辑表达式的原生支持，许多业务场景需要先做聚合运算再比较阈值。

## 一个典型的事故

2024 年某天凌晨，一个核心服务的 P99 延迟悄然攀升。运维团队配置了 `http_request_duration_seconds{quantile="0.99"} > 2` 的告警规则，但这条规则产生的大量瞬时抖动告警让值班人员疲于奔命——最终大家选择了静默该告警。而真正的问题在持续恶化，直到用户投诉才被发现。

这次事故暴露了一个核心矛盾：**我们需要更灵活的告警规则定义能力，包括"持续时长"（for 子句）、更丰富的聚合函数、以及按业务维度分组的告警策略**。而现有的配置驱动方案无法满足这些需求。

## 项目的诞生

在经历了数次类似事故后，我们决定自研一套告警引擎（alert-engine），目标明确：

1. **API 驱动的规则管理**：通过 RESTful API 实现规则的增删改查，无需重启服务。
2. **类 PromQL 表达式引擎**：自建表达式解析和求值引擎，支持选择器、聚合、比较、逻辑运算，以及常用函数（rate、increase、avg_over_time 等）。
3. **多通道告警分发**：支持 Webhook、邮件、钉钉等多种通知渠道，每条规则可独立配置。
4. **多租户支持**：规则按租户隔离，为不同业务线提供独立的告警管理能力。
5. **持续时长与状态管理**：支持 `for` 子句，区分 pending 和 firing 状态，避免瞬时抖动产生无效告警。
6. **轻量级、易部署**：Go 单二进制部署，最小依赖，适配 VictoriaMetrics 作为数据源。

我们给这个项目取名为 **alert-engine**——一个面向 VictoriaMetrics 的轻量级、可编程告警引擎。

## 技术栈选择

在项目启动之初，我们确定了以下技术选型：

| 领域 | 选择 | 理由 |
|------|------|------|
| 语言 | Go 1.23 | 团队主力语言，高性能、部署简单 |
| HTTP 框架 | gorilla/mux | 轻量、路由表达力强、社区成熟 |
| 配置格式 | YAML | 运维友好，gopkg.in/yaml.v3 支持完善 |
| UUID 生成 | google/uuid | 规则 ID 唯一性保障 |
| 数据源 | VictoriaMetrics | 已有基础设施，兼容 PromQL 查询 API |

整个项目刻意保持极简的外部依赖——仅有三个第三方库，核心逻辑全部自研。这不是出于"非此处发明"的固执，而是因为告警引擎的核心——表达式解析器和调度器——需要深度定制，现有开源方案（如 Prometheus 的规则引擎）与我们的需求存在架构层面的不匹配。

## 项目结构

```
alert-engine/
├── cmd/alert-engine/main.go     # 程序入口
├── configs/config.yaml          # 默认配置
├── internal/
│   ├── api/handler.go           # HTTP API 层
│   ├── config/config.go         # 配置加载
│   ├── dispatcher/
│   │   ├── composite.go         # 组合分发器
│   │   └── channel.go           # Webhook/邮件/钉钉通道
│   ├── engine/
│   │   ├── ast.go               # AST 节点定义
│   │   ├── evaluator.go         # 表达式求值引擎
│   │   ├── function.go          # 内置函数注册表
│   │   └── parser.go            # 词法分析+语法解析
│   ├── model/
│   │   ├── alert.go             # 告警模型
│   │   ├── config.go            # 配置模型
│   │   └── rule.go              # 规则模型
│   ├── rule/
│   │   ├── errors.go            # 哨兵错误
│   │   ├── manager.go           # 规则生命周期管理
│   │   ├── store.go             # 存储接口+内存实现
│   │   └── validator.go         # 规则校验
│   └── scheduler/
│       ├── dispatcher.go        # 分发接口
│       ├── dispatcher.go        # 调度器核心
│       └── scheduler.go         # 规则调度与状态管理
```

这是一个从痛点出发、以解决实际问题为驱动的项目。在接下来的文章中，我们将逐步展开技术调研、架构设计、核心模块实现、难题攻克的全过程。
