# 开发过程中的难题与解决

在 alert-engine 的开发过程中，我们遇到了若干技术难题。本章将这些难题逐一列出，描述问题现象、分析根本原因，并说明我们的解决方案。这些经验对于后续的迭代和维护具有重要的参考价值。

## 难题一：表达式引擎的标签匹配解析

### 问题现象

在测试表达式解析时，我们发现带有标签选择器的表达式（如 `cpu_usage{env="production"}`）无法正确解析，Parser 报错 "unexpected token"。

### 问题分析

最初的 Lexer 实现中，我们只在 `readIdent` 方法中允许标识符包含字母、数字、下划线，但标签匹配语法 `{env="production"}` 中的等号 `=` 是作为单独 Token 存在的。

我们最初的实现中，`TokEq`（等号 Token）根本没有加入 Token 类型定义，导致 Lexer 在遇到 `=` 时无法识别。

### 解决过程

1. **添加等号 Token**：
   ```go
   const (
       // ...
       TokEq  TokenKind = iota + 10  // 明确添加 =
       TokNeq                     // !=
       // ...
   )
   ```

2. **修改 Lexer 词法分析**：在 Lexer 的 Tokenize 方法中添加对 `=` 的识别：
   ```go
   case ch == '=':
       l.tokens = append(l.tokens, Token{Kind: TokEq, Value: "="})
       l.pos++
   ```

3. **修改 Parser 标签匹配解析**：确保 Parser 在解析标签时正确处理 `TokEq`：
   ```go
   func (p *Parser) parseLabelMatchers() (map[string]string, error) {
       labels := make(map[string]string)
       for p.current().Kind != TokRBrace && p.current().Kind != TokEOF {
           key := p.current().Value
           p.advance()
           if p.current().Kind != TokEq {  // 现在能识别 =
               return nil, fmt.Errorf("expected '=', got %s", p.Current().Value)
           }
           p.advance()
           labels[key] = p.current().Value  // 读取字符串值
           p.advance()
       }
       // ...
   }
   ```

### 经验总结

这是一个典型的"最小可行性实现"遗留问题——初期为了快速跑通核心语法，故意跳过了标签匹配语法的实现。后续补齐时才发现 Lexer 层面就缺少必要的 Token 类型。

> **教训**：即使是最简实现，也应该在关键语法点预留扩展接口，避免后期大幅重构。

## 难题二：VM 查询结果与表达式引擎的类型不匹配

### 问题现象

在集成测试中，表达式引擎对带范围的选择器（如 `cpu[5m]`）求值时，引擎报错 "unsupported function result type: []engine.Sample"，但函数签名返回的确实是 `[]Sample`。

### 问题分析

深入排查后发现，问题出在函数调用的返回值处理上。以 `rate()` 函数为例，它的本意是计算时间序列的变化率，返回类型应该是 `float64`（单个值），但我们的求值逻辑在某些分支错误地将 `[]Sample` 作为返回值传递。

更深层的问题在于：**函数注册表中注册的函数，其返回值类型有多种（float64、[]Sample、[]float64），求值端需要根据返回类型做不同的处理**，但我们的 switch case 分支没有覆盖 `[]Sample` 的场景。

### 解决过程

在 `evalFunction` 方法中添加对 `[]Sample` 返回类型的处理：

```go
func (e *Engine) evalFunction(ctx context.Context, node *FunctionNode) ([]Sample, error) {
    fn, ok := e.registry.Get(node.Name)
    // ... 参数求值 ...

    result, err := fn(args)
    if err != nil {
        return nil, err
    }

    // 根据返回类型做不同处理
    switch v := result.(type) {
    case float64:
        return []Sample{{Value: v}}, nil
    case []Sample:
        return v, nil  // 新增：直接返回样本序列
    case []float64:
        samples := make([]Sample, len(v))
        for i, val := range v {
            samples[i] = Sample{Value: val}
        }
        return samples, nil
    case nil:
        return []Sample{{Value: math.NaN()}}, nil
    default:
        return nil, fmt.Errorf("unsupported function result type: %T", result)
    }
}
```

### 经验总结

这是一个 **类型系统缺失** 的问题——函数返回值的类型约定没有显式表达，导致求值端和函数实现端之间存在"契约模糊"。我们通过在 switch 中显式处理每种类型来"打补丁"，更优的做法是在 FunctionRegistry 中为每个函数注册返回类型。

## 难题三：goroutine 泄漏导致服务无法优雅关闭

### 问题现象

在测试服务关闭流程时，我们发现 `scheduler.Stop()` 执行后，进程并未立即退出，而是等待了约 10 秒（规则同步间隔）后才退出。

### 问题分析

排查发现，当调度器的 `stopCh` 关闭后，`ruleWatcher` goroutine 确实退出了，但 **每个规则对应的评估 goroutine（runner）没有正确退出**。

原因在于 runner 的 `runLoop` 实现：

```go
func (s *Scheduler) runLoop(r *runner) {
    for {
        select {
        case <-r.stop:
            r.ticker.Stop()
            return
        case <-r.ticker.C:
            s.evaluateRule(r.ruleID)
        }
    }
}
```

当外部调用 `StopRule` 时，我们关闭了 `r.stop` channel，但 `evaluateRule` 正在执行一个耗时较长的 VM 查询（假设 30 秒超时），此时 **主 goroutine 已经从 `runners` map 中删除了该 runner 的记录，但 evaluateRule 仍在阻塞执行**。

更严重的是，`ruleWatcher` 在同步规则时遍历 `s.runners` 并删除不在活跃列表中的 runner——它会关闭对应 channel，但由于 `ruleWatcher` 持有读锁，而 `evaluateRule` 在求值时持有的是写锁，**锁顺序不一致导致潜在的死锁风险**（虽然在这个场景下是读写锁，不会死锁，但逻辑确实不够清晰）。

### 解决过程

1. **修复 `ruleWatcher` 中的锁粒度**：将遍历和删除操作分离，先遍历获取 ID 列表，再在锁外逐个关闭：

   ```go
   func (s *Scheduler) syncRules(ctx context.Context) {
       rules, _ := s.ruleMgr.ListActiveRules(ctx)
       activeIDs := make(map[string]bool)
       for _, r := range rules {
           activeIDs[r.ID] = true
       }

       s.mu.Lock()
       // 先记录要删除的 runner ID
       toStop := []string{}
       for id := range s.runners {
           if !activeIDs[id] {
               toStop = append(toStop, id)
           }
       }
       s.mu.Unlock()

       // 在锁外关闭
       for _, id := range toStop {
           s.StopRule(id)  // 这里会关闭 stop channel 并等待评估完成
       }
   }
   ```

2. **改进 Stop 方法的时序**：确保先关闭 stopCh（停止 ruleWatcher），再逐个关闭每个 runner：

   ```go
   func (s *Scheduler) Stop() {
       close(s.stopCh)  // 1. 停止 ruleWatcher

       s.mu.Lock()
       for _, r := range s.runners {
           close(r.stop)  // 2. 发送停止信号给每个 runner
       }
       s.mu.Unlock()

       s.runnerWg.Wait()  // 3. 等待所有 goroutine 退出
   }
   ```

### 经验总结

这个问题涉及 **Go 并发编程的两个经典陷阱**：

1. **goroutine 生命周期管理**：goroutine 启动后，外部需要通过 channel 或 context 来协调其退出时机，不能仅靠删除引用
2. **锁粒度与持锁期间的阻塞操作**：在持锁期间不应执行耗时操作（如 VM 查询），否则会阻塞其他 goroutine

正确的模式是：**快速获取/修改共享状态，然后在锁外执行耗时操作**。

## 难题四：告警重复通知

### 问题现象

在压力测试中，同一条告警在短时间内被重复发送了多次，值班人员收到了数十条重复的告警通知。

### 问题分析

经过排查，问题出在状态管理逻辑中。我们最初的 `handleAlerts` 实现只处理了"首次触发"的情况，但没有正确处理"持续触发"的去重。

更深层的原因在于 **alert ID 的生成策略**：我们使用 `ruleID + labelsKey(sample.Labels)` 作为告警 ID，这在标签完全一致时能够正确识别为同一个告警。但如果标签中包含时间戳相关字段（或者字段值在每次查询时略有不同），就会产生不同的告警 ID，导致重复通知。

另一个原因是我们**没有记录上次通知的时间**，每次状态变化都会重新通知。

### 解决过程

1. **改进 alert ID 生成**：检查标签中是否有不稳定字段（如时间戳），如果有则排除在 key 生成之外：

   ```go
   func labelsKey(labels map[string]string) string {
       if len(labels) == 0 {
           return ""
       }
       // 过滤掉不稳定的时间相关标签
       excludeKeys := map[string]bool{"__name__": true, "timestamp": true}
       
       var parts []string
       for k, v := range labels {
           if excludeKeys[k] {
               continue
           }
           parts = append(parts, k+"="+v)
       }
       sort.Strings(parts)  // 排序保证顺序一致
       return strings.Join(parts, ",")
   }
   ```

2. **添加通知时间戳检查**：在 `handleAlerts` 中，增加最小通知间隔判断：

   ```go
   const MinNotifyInterval = 5 * time.Minute

   for id, alert := range newFiring {
       if _, wasFiring := oldFiring[id]; !wasFiring {
           // 新触发，检查上次通知时间
           if state.lastNotifyTime.Add(MinNotifyInterval).After(time.Now()) {
               continue  // 跳过，在最小通知间隔内
           }
           // 发送通知...
       }
   }
   ```

3. **持久化 alert 状态**：这是长期解决方案——将告警状态写入持久化存储（数据库），重启后可以恢复状态，避免重启后的"首评告警"重复通知。

### 经验总结

告警重复通知是一个**看似简单、实则复杂**的问题，涉及到：

- **ID 生成的稳定性**：必须确保相同告警实体生成相同的 ID
- **去重策略**：需要明确"什么算同一条告警"
- **通知频率控制**：需要实现最小通知间隔或通知聚合
- **状态持久化**：服务重启后的状态恢复

我们的临时方案（ID 稳定性优化 + 最小通知间隔）缓解了问题，但要从根本上解决，需要引入持久化存储。

## 难题五：正则标签匹配的缺失

### 问题现象

用户期望使用正则表达式来匹配标签，如 `cpu_usage{env=~"prod.*"}`，但系统报错 "unexpected character: ~"。

### 问题分析

我们的 Lexer 只实现了精确匹配（`=`），没有实现正则匹配运算符（`=~` 表示"匹配"，`!~` 表示"不匹配"）。

这是 **PromQL 兼容性问题** 之一——虽然正则匹配在日常告警中使用频率不如精确匹配高，但在某些场景下（如按服务名正则分组、按环境批量匹配）是刚需。

### 解决过程

1. **添加 Token**：
   ```go
   TokRegexMatch  // =~
   TokRegexNotMatch  // !~
   ```

2. **修改 Lexer**：识别 `=~` 和 `!~`：
   ```go
   case ch == '=' && l.peek(1) == '~':
       l.tokens = append(l.tokens, Token{Kind: TokRegexMatch, Value: "=~"})
       l.pos += 2
   case ch == '!' && l.peek(1) == '=':
       l.tokens = append(l.tokens, Token{Kind: TokNeq, Value: "!="})
       l.pos += 2
   ```

3. **修改 Parser**：在 `parseLabelMatchers` 中处理正则匹配：
   ```go
   for p.current().Kind != TokRBrace && p.current().Kind != TokEOF {
       key := p.current().Value
       p.advance()
       
       // 匹配多种运算符
       var matchType string
       switch p.current().Kind {
       case TokEq:
           matchType = "exact"
       case TokRegexMatch:
           matchType = "regex"
       case TokRegexNotMatch:
           matchType = "not_regex"
       default:
           return nil, fmt.Errorf("expected label matcher operator")
       }
       // ...
   }
   ```

4. **修改 Evaluator**：在标签过滤时使用正则匹配：
   ```go
   func filterByLabelsRegex(samples []Sample, labels map[string]string) []Sample {
       if len(labels) == 0 {
           return samples
       }
       // 对每个样本的标签进行正则匹配
       // ...
   }
   ```

### 当前状态

截至当前版本，正则匹配功能尚未完全实现——AST 节点定义了字段，但 `filterByLabels` 函数中只实现了精确匹配逻辑。正则匹配是后续迭代的高优先级需���。

### 经验总结

**功能的优先级排序至关重要**。正则匹配在功能列表上，但不是"最小可行性产品"所必需的。明确 MVP 边界，控制功能范围，能够显著加速项目交付——但需要文档化未支持的功能，避免用户误用。

## 难题六：并发更新导致规则版本冲突

### 问题现象

在并发测试中，两个管理员同时修改同一条规则，后提交的管理员的修改被静默覆盖，没有收到任何错误提示。

### 问题分析

这违反了数据一致性原则。根本原因是我们**缺少乐观锁机制**——更新规则时没有检查版本号，直接覆盖。

### 解决过程

1. **在 Rule 模型中添加 Version 字段**（已完成）：
   ```go
   type Rule struct {
       // ...
       Version     int64
       // ...
   }
   ```

2. **在 Manager.UpdateRule 中添加版本检查**：
   ```go
   func (m *Manager) UpdateRule(ctx context.Context, r *model.Rule) error {
       existing, err := m.store.Get(ctx, r.ID)
       if err != nil {
           return err
       }
       
       // 关键：版本检查
       if existing.Version != r.Version {
           return ErrVersionConflict
       }
       
       // 更新版本号
       r.Version = existing.Version + 1
       r.UpdatedAt = time.Now()
       
       return m.store.Update(ctx, r)
   }
   ```

3. **定义冲突错误**：
   ```go
   var ErrVersionConflict = errors.New("version conflict")
   ```

### 经验总结

并发更新冲突是分布式系统的经典问题。我们的解决方案是 **乐观锁**——通过版本号检测冲突。这在冲突概率较低的场景下非常高效。

如果冲突概率较高（如高频繁操作），则应考虑 **悲观锁**（SELECT FOR UPDATE）或分布式锁。

## 总结

开发过程中的难题按照性质可分类如下：

| 类别 | 难题 | 解决方案 |
|------|------|----------|
| 语法完整性 | 标签匹配等号缺失 | 补齐 Token 和 Parser 分支 |
| 类型系统 | 函数返回类型不匹配 | 扩展 switch case 处理 |
| 并发安全 | goroutine 泄漏、状态不一致 | 改进生命周期管理和锁粒度 |
| 数据一致性 | 规则并发更新覆盖 | 引入乐观锁 |
| 功能完整性 | 正则标签匹配缺失 | 预留扩展点，分期实现 |
| 业务逻辑 | 告警重复通知 | ID 稳定性优化 + 最小通知间隔 |

这些难题的共同特点是：**它们只在集成测试或生产环境中暴露**，单元测试难以覆盖。这提醒我们：

1. **端到端集成测试不可或缺**——必须验证完整的数据流
2. **边界条件要尽早测试**——空值、并发、极端输入
3. **MVP 阶段要明确边界**——清晰文档化未支持的功能
