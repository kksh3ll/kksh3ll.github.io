# 项目总结与未来展望

## 项目回顾

历时数月的开发，alert-engine 已从一个解决内部痛点的内部工具，成长为一个具备生产就绪能力的轻量级告警引擎。回顾整个项目历程，我们不仅交付了一个可用的系统，更在这个过程中积累了一笔可观的 技术财富。

### 核心交付

alert-engine 目前已具备以下核心能力：

1. **API 驱动的告警规则管理**
   - 通过 RESTful API 实现规则的增删改查、启停控制
   - 规则变更无需重启服务，秒级生效
   - 多租户规则隔离能力

2. **自研表达式引擎**
   - 支持类 PromQL 表达式语法
   - 覆盖选择器、聚合、比较、逻辑运算
   - 内置 30+ 常用函数（rate、increase、avg_over_time 等）
   - 完整的词法分析和递归下降解析器

3. **灵活的调度与状态管理**
   - 每规则独立评估 goroutine，支持不同评估间隔
   - 状态 diff 通知模型，避免告警风暴
   - 优雅的 goroutine 生命周期管理

4. **多通道告警分发**
   - Webhook（通用 HTTP 回调）
   - 邮件（SMTP + TLS 支持）
   - 钉钉（Webhook + 签名验证）
   - 每规则独立��置通知渠道

5. **极简部署**
   - Go 单二进制
   - 仅有 3 个外部依赖
   - YAML 配置文件

### 技术指标

| 指标 | 数值 |
|------|------|
| 代码行数（不含测试） | ~3500 行 |
| 外部依赖 | 3 个（gorilla/mux, yaml, uuid） |
| 核心模块数 | 7 个 |
| 单元测试覆盖率 | ~45%（核心模块） |
| 启动时间 | < 1 秒 |
| 评估延迟（无 VM 延迟） | < 50ms（典型表达式） |

### 架构设计亮点

1. **接口抽象，分层解耦**：各层通过接口定义能力，具体实现可替换
2. **依赖注入，组合装配**：通过构造函数接收依赖，便于测试和 mock
3. **渐进式构建**：先用最小可行实现验证核心链路，再逐步补齐生产级能力
4. **运行时动态性**：所有核心操作通过 API 动态完成

## 项目不足与局限

坦诚地说，当前版本的 alert-engine 仍有许多不足：

1. **PromQL 兼容性有限**：约 80% 覆盖率，部分高级语法（子查询、正则标签匹配、部分函数）未支持
2. **存储无持久���**：规则和告警状态在内存中，重启后丢失
3. **通知异步缺失**：当前所有通知同步发送，可能阻塞评估链路
4. **缺少 for 子句**：当前不支持"持续 N 秒才触发"的判断
5. **监控可观测性不足**：缺少 metrics 导出，运行时状态不可见
6. **高可用能力缺失**：单实例部署，无多副本水平扩展

这些局限并非"开发事故"，而是 **有意识的 MVP 边界选择**——我们优先交付核心链路，确保可用，再逐步补齐生产级能力。

## 技术负债

以下技术债务需要在后续迭代中偿还：

| 债务项 | 影响 | 优先级 |
|--------|------|--------|
| 无持久化存储 | 重启后规则丢失，告警历史不可查 | 高 |
| 无 metrics 监控 | 无法感知系统健康状态 | 高 |
| 无 for 子句实现 | 无法抑制瞬时抖动告警 | 中 |
| 并发调度模型单一 | 千规则以上时 goroutine 开销显著 | 中 |
| 通知模板硬编码 | 无法自定义告警消息格式 | 低 |
| 正则标签匹配缺失 | 部分业务场景无法满足 | 低 |

## 未来展望

下一阶段，alert-engine 将向 **生产就绪** 目标演进，重点方向如下：

### 一、持久化存储

引入数据库存储（SQLite 作为默认，PostgreSQL/MySQL 作为可选），实现规则和告警状态的持久化。这是生产部署的前提条件。

**关键任务**：
- 定义 Store 接口的 SQL 实现
- 编写数据库迁移脚本
- 实现规则���告警的增删改查

### 二、For 子句支持

完善 `ForSpec` 的评估逻辑，实现"持续时长"判断。只有当条件持续满足 N 秒后才触发告警，这将大幅减少瞬时抖动告警。

**关键任务**：
- 在 alertState 中记录条件首次满足的时间戳
- 在 evaluateRule 中增加持续时长判断
- 添加 pending 状态的告警事件（可选）

### 三、可观测性

引入 Prometheus metrics 导出，监控引擎运行状态。

**关键任务**：
- 暴露评估次数、评估延迟、告警触发数等指标
- 暴露调度器队列长度、规则数等运行时状态
- 集成到现有监控体系

### 四、高级告警功能

1. **告警抑制（Inhibition）**：某些告警触发时，自动抑制关联告警
2. **告警静默（Silences）**：支持临时静默某类告警
3. **告警聚合**：同一规则短时间内多次触发，聚合为一条通知
4. **告警升级**：告警持续未响应，自动升级通知渠道

### 五、分布式扩展

当前单实例模型无法支撑大规模规则评估。后续可考虑：

1. **评估器池化**：将规则按租户或分组分配到多个 worker 评估
2. **调度器选举**：多实例部署时，通过一致性算法（如 Raft）选主
3. **状态共享**：评估状态外置到 Redis/etcd，支持多副本

### 六、生态集成

1. **多数据源支持**：除了 VictoriaMetrics，可接入 Prometheus、Thanos、InfluxDB
2. **配置中心集成**：对接 Nacos、Apollo 等配置中心，实现规则的动态下发
3. **日志集成**：结构化日志（slog），支持日志收集和分析
