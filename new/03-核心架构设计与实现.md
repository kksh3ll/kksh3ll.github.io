# 核心架构设计与实现

## 整体架构

alert-engine 采用分层架构设计，从上到下依次为：

1. **API 层**（`internal/api`）：接收 HTTP 请求，提供规则的 CRUD、启停控制、查询等接口。
2. **规则管理层**（`internal/rule`）：规则的校验、版本管理、生命周期操作。底层通过 `Store` 接口对接不同存储实现。
3. **调度层**（`internal/scheduler`）：规则评估的调度中心，管理每个规则的评估 goroutine，负责状态管理和告警事件分发。
4. **表达式引擎层**（`internal/engine`）：自建的类 PromQL 表达式解析和求值引擎。
5. **数据源层**（`internal/vm`）：VictoriaMetrics HTTP API 客户端，负责查询时序数据。
6. **通知分发层**（`internal/dispatcher`）：多通道告警分发器，支持 Webhook、邮件、钉钉。

```
┌─────────────────────────────────────────────────────────────┐
│                         API Layer                           │
│              (gorilla/mux Routes & Handlers)               │
└─────────────────────┬───────────────────────────────────────┘
                      │
                      ▼
┌─────────────────────────────────────────────────────────────┐
│                    Rule Manager Layer                       │
│         (Validate, Create, Update, Delete, CRUD)           │
└─────────────────────┬───────────────────────────────────────┘
                      │
                      ▼
┌─────────────────────────────────────────────────────────────┐
│                     Scheduler Layer                         │
│    (Schedule, Evaluate, State Management, Dispatch)        │
└──────┬──────────────────┬────────────────────┬─────────────┘
       │                  │                    │
       ▼                  ▼                    ▼
┌─────────────┐   ┌─────────────┐    ┌─────────────────────┐
│   Engine    │   │  VM Client  │    │    Dispatcher       │
│  (Parse &   │◄──│  (Query     │    │  (Webhook/Email/    │
│   Evaluate) │   │   API)      │    │   DingTalk)         │
└─────────────┘   └─────────────┘    └─────────────────────┘
```

各层之间通过 **接口** 解除耦合，而非直接依赖具体实现。这使得：

- `Store` 可以是内存存储，也可以是未来的 PostgreSQL/MySQL 存储
- `VMClient` 可以是 VictoriaMetrics，也可以通过适配器支持其他数据源
- `Dispatcher` 可以按需扩展新通道

## 核心模块设计

### 1. 规则模型（Rule Model）

规则是整个系统的核心实体。参考 Prometheus 告警规则语法，我们定义了如下字段：

```go
type Rule struct {
    ID          string                // 唯一标识，UUID
    Name        string                // 规则名称
    Expr        string                // 表达式（类 PromQL）
    For         *ForSpec              // 持续时长
    Severity    Severity              // 严重级别：critical/warning/info
    Labels      map[string]string     // 标签
    Annotations map[string]string     // 注释（告警描述等）
    Channels    []NotificationChannel // 通知渠道配置
    State       RuleState             // 规则状态：active/paused/deleted
    Version     int64                 // 乐观锁版本号
    TenantID    string                // 多租户隔离
    Interval    time.Duration         // 评估间隔
    CreatedAt   time.Time
    UpdatedAt   time.Time
}
```

关键设计点：

- **乐观锁**：通过 `Version` 字段防止并发更新冲突。更新时要求客户端传入期望版本号，服务端对比后拒绝陈旧请求。
- **通知渠道内嵌**：每条规则独立配置通知渠道，而非全局统一配置。这是最核心的差异化能力。
- **多租户**：`TenantID` 字段实现租户隔离，List/ListActive 接口支持按租户过滤。

### 2. 告警模型（Alert Model）

告警是规则触发后的产物：

```go
type Alert struct {
    ID           string            // alert-xxxx
    RuleID       string            // 关联规则 ID
    RuleName     string            // 关联规则名称
    State        AlertState        // firing/pending/resolved
    Severity     Severity
    Labels       map[string]string // 规则标签 + 查询返回的时序标签
    Annotations  map[string]string
    Value        float64           // 当前值
    Threshold    float64           // 阈值（待扩展）
    Expr         string            // 原始表达式
    EvaluatedAt  time.Time         // 本次评估时间
    FiredAt      *time.Time        // 首次触发时间
    ResolvedAt   *time.Time        // 解决时间
    LastNotified *time.Time        // 上次通知时间
}
```

告警的状态流转：

```
         ┌─────────┐
         │  None   │ (无告警产生)
         └────┬────┘
              │
    ┌─────────┴─────────┐
    │  Expression > 0   │ (条件满足)
    ▼                   ▼
┌─────────┐        ┌──────────┐
│ Pending │───────►│  Firing  │ (持续 for 时长后)
└─────────┘        └────┬─────┘
                        │
               ┌────────┴────────┐
               │ Expression <= 0 │ (条件不再满足)
               ▼                 ▼
          ┌──────────┐      ┌─────────┐
          │ Resolved │      │ Firing  │ (继续 firing)
          └──────────┘      └─────────┘
```

### 3. 数据源抽象（VictoriaMetrics Client）

`VMClient` 是整个系统的数据源头。它封装了对 VictoriaMetrics `/query` 和 `/query_range` 两个 HTTP API 的调用：

```go
type VMClient interface {
    Query(ctx context.Context, expr string, start, end time.Time, step time.Duration) (*engine.QueryResult, error)
}
```

返回的 `QueryResult` 包含一组 `Sample`：

```go
type Sample struct {
    Labels map[string]string // 时序标签
    Value  float64           // 数值
    Time   time.Time         // 时间戳
}
```

这个抽象使得表达式引擎只需调用统一的 Query 方法，无需关心底层是 VM 还是 Prometheus。后续如需支持其他数据源，只需实现新的 Client 并注入即可。

### 4. 存储抽象（Store Interface）

存储层通过接口抽象，便于扩展：

```go
type Store interface {
    Create(ctx context.Context, rule *model.Rule) error
    Update(ctx context.Context, rule *model.Rule) error
    Delete(ctx context.Context, id string) error
    Get(ctx context.Context, id string) (*model.Rule, error)
    List(ctx context.Context, tenantID string) ([]*model.Rule, error)
    ListActive(ctx context.Context) ([]*model.Rule, error)
}
```

当前实现 `MemoryStore` 是内存 Map + sync.RWMutex。后续可扩展为持久化存储（SQLite / PostgreSQL / etcd），只需实现同一接口。

### 5. 分发抽象（Dispatcher Interface）

告警通知通过 Dispatcher 接口解耦：

```go
type Dispatcher interface {
    Notify(ctx context.Context, alert model.Alert, channels []model.NotificationChannel) error
}
```

`NotificationChannel` 定义了告警发送目标：

```go
type NotificationChannel struct {
    Type   string            // webhook / email / dingtalk
    Config map[string]string // 渠道-specific 配置
}
```

这种设计让每条规则可以独立配置多个通知渠道，渠道配置作为规则的一部分持久化存储。

## 主入口与组件组装

main.go 是整个系统的组装点，展示了各组件的依赖关系：

```go
func main() {
    cfg := config.LoadOrDefault(*configPath)

    // 1. 存储层：内存存储（后续可换持久化）
    store := rule.NewMemoryStore()
    ruleMgr := rule.NewManager(store)

    // 2. 数据源层：VictoriaMetrics 客户端
    vmClient := vm.NewClient(cfg.VM.Addr, cfg.VM.Tenant, time.Duration(cfg.VM.Timeout)*time.Second)

    // 3. 表达式引擎层
    eng := engine.NewEngine(vmClient.Query)

    // 4. 分发层：多通道分发器
    dispatcher := dispatcher.NewChannelDispatcher(
        &dispatcher.WebhookConfig{...},
        &dispatcher.EmailConfig{...},
        &dispatcher.DingTalkConfig{...},
    )

    // 5. 调度层：规则调度与评估
    sched := scheduler.NewScheduler(ruleMgr, eng, vmClient, dispatcher, cfg.Scheduler.Workers)
    sched.Start(context.Background())

    // 6. API 层
    handler := api.NewHandler(ruleMgr, sched, vmClient)
    router := mux.NewRouter()
    handler.RegisterRoutes(router)

    // 7. HTTP 服务器
    srv := &http.Server{Addr: cfg.Server.Addr, Handler: router}
    go srv.ListenAndServe()

    // 8. 信号处理与优雅 shutdown
    signal.Notify(quit, syscall.SIGINT, syscall.SIGTERM)
    <-quit
    sched.Stop()
    srv.Shutdown(ctx)
}
```

这是一个经典的 **依赖注入** 模式——各组件通过构造函数接收依赖，而非在组件内部 new 出来。这带来了极佳的可测试性：我们可以在单元测试中 mock VMClient、mock Dispatcher，注入到 Engine 和 Scheduler 中进行独立测试。

## API 路由设计

系统暴露了以下 RESTful 端点：

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/health` | 健康检查 |
| GET | `/api/v1/rules` | 列出规则（支持 tenant_id 过滤） |
| POST | `/api/v1/rules` | 创建规则 |
| GET | `/api/v1/rules/{id}` | 获取单条规则 |
| PUT | `/api/v1/rules/{id}` | 更新规则 |
| DELETE | `/api/v1/rules/{id}` | 删除规则 |
| POST | `/api/v1/rules/{id}/pause` | 暂停规则 |
| POST | `/api/v1/rules/{id}/resume` | 恢复规则 |
| POST | `/api/v1/rules/{id}/start` | 手动启动规则评估 |
| POST | `/api/v1/rules/{id}/stop` | 手动停止规则评估 |
| GET | `/api/v1/alerts` | 列出告警（预留） |
| GET | `/api/v1/query` | 表达式即时查询 |
| GET | `/api/v1/query_range` | 表达式范围查询 |

通过 API，运维人员可以在运行时动态管理告警规则，无需修改配置文件、无需重启服务。

## 配置管理

系统通过 YAML 文件配置：

```yaml
server:
  addr: ":8080"

vm:
  addr: "http://localhost:8428"  # VictoriaMetrics 地址
  tenant: ""                      # 多租户 tenant ID
  timeout: 30                     # 查询超时（秒）

scheduler:
  workers: 4                      # 评估并发度
  eval_timeout: 30                # 单次评估超时（秒）

store:
  type: "memory"                 # 存储类型（可扩展为 sqlite/postgres）
  dsn: ""                        # 连接字符串
```

默认配置通过 `config.Default()` 提供，程序启动时先尝试加载指定路径的配置，若失败则回退到默认值。

## 架构原则总结

1. **接口抽象，分层解耦**：每层通过接口定义能力，具体实现可替换。
2. **依赖注入，组合装配**：组件通过构造函数接收依赖，便于测试和替换。
3. **领域模型驱动**：Rule、Alert、NotificationChannel 等领域实体是系统核心，围绕它们构建业务逻辑。
4. **渐进式构建**：先用最小可行实现（内存存储、同步调度）验证核心链路，后续迭代补齐持久化、分布式等能力。
5. **运行时动态性**：所有核心操作（规则增删改、启停控制）均通过 API 动态完成，无需重启。
