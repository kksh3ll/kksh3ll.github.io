# 多通道告警分发设计

告警的最终价值在于**通知**——只有将告警信息及时、准确地送达相关人员，问题才能得到响应和解决。本章将深入 alert-engine 的告警分发模块，探讨其如何支持 Webhook、邮件、钉钉等多种通知渠道，以及如何通过接口抽象实现灵活扩展。

## 分发器职责与设计目标

告警分发器（Dispatcher）承担以下职责：

1. **多渠道支持**：支持 Webhook、邮件、钉钉等通知渠道，未来可扩展企业微信、短信、Slack 等
2. **灵活配置**：每条规则可独立配置通知渠道，渠道配置作为规则的一部分存储
3. **可靠投递**：支持重试机制，避免因网络抖动导致告警丢失
4. **消息格式化**：根据不同渠道的特性，生成符合其规范的告警消息格式
5. **安全签名**：支持钉钉等渠道的消息签名验证，防止伪造

## 分发器接口设计

为了支持多渠道和便于扩展，我们定义了清晰的接口：

```go
type Dispatcher interface {
    Notify(ctx context.Context, alert model.Alert, channels []model.NotificationChannel) error
}
```

`NotificationChannel` 定义了通道类型和配置：

```go
type NotificationChannel struct {
    Type   string            // channel type: webhook / email / dingtalk
    Config map[string]string // channel-specific config
}
```

这种设计的核心思想是：**Channel 配置实例化，Dispatcher 根据配置动态选择发送方式**。

## ChannelDispatcher 实现

`ChannelDispatcher` 是当前的默认实现，支持 Webhook、邮件、钉钉三种渠道：

```go
type ChannelDispatcher struct {
    webhookCfg *WebhookConfig
    emailCfg   *EmailConfig
    dingCfg    *DingTalkConfig
    httpClient *http.Client
}
```

各渠道的配置通过构造函数注入：

```go
func NewChannelDispatcher(
    webhookCfg *WebhookConfig,
    emailCfg *EmailConfig,
    dingCfg *DingTalkConfig,
) *ChannelDispatcher {
    client := &http.Client{
        Timeout: webhookCfg.Timeout,
    }
    return &ChannelDispatcher{
        webhookCfg: webhookCfg,
        emailCfg:   emailCfg,
        dingCfg:    dingCfg,
        httpClient: client,
    }
}
```

### Notify 方法入口

```go
func (d *ChannelDispatcher) Notify(
    ctx context.Context,
    alert model.Alert,
    channels []model.NotificationChannel,
) error {
    if len(channels) == 0 {
        return nil
    }

    for _, ch := range channels {
        switch ch.Type {
        case "webhook":
            if err := d.sendWebhook(ctx, alert, ch.Config); err != nil {
                return err
            }
        case "email":
            if err := d.sendEmail(ctx, alert, ch.Config); err != nil {
                return err
            }
        case "dingtalk":
            if err := d.sendDingTalk(ctx, alert, ch.Config); err != nil {
                return err
            }
        default:
            return fmt.Errorf("unsupported channel type: %s", ch.Type)
        }
    }
    return nil
}
```

这里的策略是 **fail-fast**：只要有一个渠道发送失败，就立即返回错误。这并非最优策略（更优的做法是记录失败渠道并重试），但对于第一版实现来说是合理的选择。

## 一、Webhook 通知

Webhook 是最通用的通知方式——将告警 JSON 推送到指定的 HTTP 端点。

### Webhook 配置

```go
type WebhookConfig struct {
    URL        string            // 全局默认 Webhook URL
    Method     string            // HTTP 方法，默认 POST
    Headers    map[string]string // 自定义请求头
    Secret     string            // 签名密钥，用于 HMAC-SHA256 签名
    Timeout    time.Duration     // 请求超时
    RetryCount int               // 重试次数
    RetryDelay time.Duration     // 重试间隔
}
```

### sendWebhook 实现

```go
func (d *ChannelDispatcher) sendWebhook(
    ctx context.Context,
    alert model.Alert,
    config map[string]string,
) error {
    // 1. 解析 URL（支持覆盖全局配置）
    webhookURL := d.webhookCfg.URL
    if url, ok := config["url"]; ok {
        webhookURL = url
    }
    if webhookURL == "" {
        return fmt.Errorf("webhook url is empty")
    }

    // 2. 解析 Method
    method := d.webhookCfg.Method
    if m, ok := config["method"]; ok {
        method = m
    }
    if method == "" {
        method = "POST"
    }

    // 3. 构造请求体
    body, err := json.Marshal(map[string]interface{}{
        "alert": alert,
    })
    if err != nil {
        return fmt.Errorf("marshal alert: %w", err)
    }

    // 4. 创建请求
    req, err := http.NewRequestWithContext(ctx, method, webhookURL, bytes.NewReader(body))
    // ...

    req.Header.Set("Content-Type", "application/json")
    // 添加全局自定义头
    for k, v := range d.webhookCfg.Headers {
        req.Header.Set(k, v)
    }
    // 添加通道级自定义头
    if header, ok := config["headers"]; ok {
        var headers map[string]string
        json.Unmarshal([]byte(header), &headers)
        for k, v := range headers {
            req.Header.Set(k, v)
        }
    }

    // 5. 签名（如果配置了 Secret）
    if d.webhookCfg.Secret != "" {
        mac := hmac.New(sha256.New, []byte(d.webhookCfg.Secret))
        mac.Write(body)
        signature := hex.EncodeToString(mac.Sum(nil))
        req.Header.Set("X-Signature", signature)
    }

    // 6. 重试循环
    var lastErr error
    retryCount := d.webhookCfg.RetryCount
    retryDelay := d.webhookCfg.RetryDelay
    if retryDelay == 0 {
        retryDelay = time.Second
    }

    for i := 0; i <= retryCount; i++ {
        resp, err := d.httpClient.Do(req)
        if err != nil {
            lastErr = err
            time.Sleep(retryDelay)
            continue
        }
        resp.Body.Close()

        if resp.StatusCode >= 200 && resp.StatusCode < 300 {
            return nil
        }
        lastErr = fmt.Errorf("http status: %d", resp.StatusCode)
        time.Sleep(retryDelay)
    }

    return lastErr
}
```

### 设计要点

1. **全局配置 + 通道覆盖**：全局配置作为默认值，规则级别可单独指定 URL、Method、Headers
2. **HMAC 签名**：如果配置了 Secret，会在请求头中添加 `X-Signature`，接收方可用于验签
3. **指数退避（可选）**：当前实现是固定间隔重试，可改进为指数退避

## 二、邮件通知

邮件是传统的告警通知方式，至今仍被广泛使用。

### 邮件配置

```go
type EmailConfig struct {
    SMTPHost string // SMTP 服务器地址
    SMTPPort int    // 端口（25/465/587）
    Username string // 用户名
    Password string // 密码
    From     string // 发件人
    To       string // 默认收件人
    UseTLS   bool   // 是否使用 TLS
}
```

### sendEmail 实现

```go
func (d *ChannelDispatcher) sendEmail(
    ctx context.Context,
    alert model.Alert,
    config map[string]string,
) error {
    // 1. 解析配置（支持覆盖）
    smtpHost := d.emailCfg.SMTPHost
    if h, ok := config["smtp_host"]; ok {
        smtpHost = h
    }
    // ... 其他字段解析 ...

    if smtpHost == "" || to == "" {
        return fmt.Errorf("email config incomplete")
    }

    // 2. 构造邮件内容
    subject := fmt.Sprintf("[%s] %s - %s", alert.Severity, alert.State, alert.RuleName)
    body := fmt.Sprintf(`
Alert: %s
Rule: %s
Severity: %s
State: %s
Value: %.2f
Time: %s
Labels: %v
Annotations: %v
Expression: %s
`, alert.ID, alert.RuleName, alert.Severity, alert.State, alert.Value,
       alert.EvaluatedAt, alert.Labels, alert.Annotations, alert.Expr)

    msg := []byte(fmt.Sprintf("From: %s\r\nTo: %s\r\nSubject: %s\r\n\r\n%s",
        from, to, subject, body))

    // 3. 发送邮件
    auth := smtp.PlainAuth("", username, password, smtpHost)
    addr := fmt.Sprintf("%s:%d", smtpHost, smtpPort)

    var err error
    if d.emailCfg.UseTLS {
        err = sendMailTLS(addr, auth, from, []string{to}, msg)
    } else {
        err = smtp.SendMail(addr, auth, from, []string{to}, msg)
    }

    return err
}
```

### TLS 邮件发送

标准库的 `smtp.SendMail` 不支持 SSL/TLS（端口 465），需要自行实现 TLS 连接：

```go
func sendMailTLS(addr string, auth smtp.Auth, from string, to []string, msg []byte) error {
    conn, err := tlsConnect(addr)
    if err != nil {
        return err
    }
    defer conn.Close()

    c, err := smtp.NewClient(conn, strings.Split(addr, ":")[0])
    if err != nil {
        return err
    }
    defer c.Quit()

    if err = c.Auth(auth); err != nil {
        return err
    }
    if err = c.Mail(from); err != nil {
        return err
    }
    for _, addr := range to {
        if err = c.Rcpt(addr); err != nil {
            return err
        }
    }
    w, err := c.Data()
    _, err = w.Write(msg)
    err = w.Close()
    return err
}

func tlsConnect(addr string) (net.Conn, error) {
    host := strings.Split(addr, ":")[0]
    port := "465"
    if strings.Contains(addr, ":") {
        port = strings.Split(addr, ":")[1]
    }
    conn, err := tls.Dial("tcp", host+":"+port, &tls.Config{ServerName: host})
    return conn, err
}
```

### 设计要点

1. **明文密码风险**：当前将 SMTP 密码以明文存储在配置中，生产环境应使用密钥管理服务（KMS）
2. **邮件格式**：纯文本格式，结构简单但信息完整
3. **TLS 支持**：显式区分 25 端口（无 TLS）和 465 端口（SSL/TLS）

## 三、钉钉通知

钉钉是企业内部最常用的即时通讯工具，其机器人 Webhook 机制允许自定义消息推送。

### 钉钉配置

```go
type DingTalkConfig struct {
    WebhookURL string // 全局默认 Webhook URL
    Secret     string // 加签密钥
    AtMobiles  string // @手机号列表，逗号分隔
    IsAtAll    bool   // @所有人
}
```

### sendDingTalk 实现

```go
func (d *ChannelDispatcher) sendDingTalk(
    ctx context.Context,
    alert model.Alert,
    config map[string]string,
) error {
    // 1. 解析配置
    webhookURL := d.dingCfg.WebhookURL
    if w, ok := config["webhook_url"]; ok {
        webhookURL = w
    }
    if webhookURL == "" {
        return fmt.Errorf("dingtalk webhook url is empty")
    }

    secret := d.dingCfg.Secret
    if s, ok := config["secret"]; ok {
        secret = s
    }

    // 2. 构建消息体（Markdown 格式）
    var msg map[string]interface{}
    if secret != "" {
        // 3. 签名验证：按钉钉规则生成签名
        timestamp := time.Now().UnixMilli()
        sign := generateDingTalkSign(secret, timestamp)
        msg = map[string]interface{}{
            "msgtype": "markdown",
            "markdown": map[string]string{
                "title": fmt.Sprintf("[%s] %s", alert.Severity, alert.RuleName),
                "text":  formatDingTalkMessage(alert),
            },
            "at": map[string]interface{}{
                "atMobiles": strings.Split(d.dingCfg.AtMobiles, ","),
                "isAtAll":   d.dingCfg.IsAtAll,
            },
            "timestamp": timestamp,
            "sign":      sign,
        }
    } else {
        msg = map[string]interface{}{
            "msgtype": "markdown",
            "markdown": map[string]string{
                "title": fmt.Sprintf("[%s] %s", alert.Severity, alert.RuleName),
                "text":  formatDingTalkMessage(alert),
            },
            "at": map[string]interface{}{
                "atMobiles": strings.Split(d.dingCfg.AtMobiles, ","),
                "isAtAll":   d.dingCfg.IsAtAll,
            },
        }
    }

    // 4. 发送请求
    body, _ := json.Marshal(msg)
    req, err := http.NewRequestWithContext(ctx, "POST", webhookURL, bytes.NewReader(body))
    req.Header.Set("Content-Type", "application/json")

    resp, err := d.httpClient.Do(req)
    defer resp.Body.Close()

    if resp.StatusCode != 200 {
        return fmt.Errorf("dingtalk response status: %d", resp.StatusCode)
    }

    return nil
}
```

### 钉钉签名算法

钉钉的加签机制基于 HMAC-SHA256：

```go
func generateDingTalkSign(secret string, timestamp int64) string {
    stringToSign := fmt.Sprintf("%d\n%s", timestamp, secret)
    h := hmac.New(sha256.New, []byte(secret))
    h.Write([]byte(stringToSign))
    return hex.EncodeToString(h.Sum(nil))
}
```

### 消息格式化

```go
func formatDingTalkMessage(alert model.Alert) string {
    labels := ""
    for k, v := range alert.Labels {
        labels += fmt.Sprintf("- %s: %s\n", k, v)
    }
    return fmt.Sprintf("## %s\n\n- **Severity**: %s\n- **State**: %s\n- **Value**: %.2f\n- **Time**: %s\n\n### Labels\n%s\n\n### Annotations\n%s\n",
        alert.RuleName,
        alert.Severity,
        alert.State,
        alert.Value,
        alert.EvaluatedAt.Format(time.RFC3339),
        labels,
        formatAnnotations(alert.Annotations),
    )
}
```

钉钉 Markdown 支持有限的格式，我们选取了最关键的信息进行展示。

### 设计要点

1. **签名验证**：钉钉 Webhook 支持加签模式，安全性更高
2. **@功能**：支持 @指定人员或 @所有人
3. **Markdown 格式**：比纯文本更美观，信息层次更清晰

## 四、组合分发器

除了单渠道分发器，我们还提供了 `CompositeDispatcher` 和 `MultiDispatcher`，用于更复杂的分发场景。

### CompositeDispatcher

按通道类型（webhook/email/dingtalk）路由到不同的分发器：

```go
type CompositeDispatcher struct {
    channels map[string]*ChannelDispatcher
}

func (d *CompositeDispatcher) Notify(
    ctx context.Context,
    alert model.Alert,
    channels []model.NotificationChannel,
) error {
    for _, ch := range channels {
        if chd, ok := d.channels[ch.Type]; ok {
            if err := chd.Notify(ctx, alert, []model.NotificationChannel{ch}); err != nil {
                return err
            }
        }
    }
    return nil
}
```

### MultiDispatcher

按分发器列表顺序执行：

```go
type MultiDispatcher struct {
    dispatchers []Dispatcher
}

func (m *MultiDispatcher) Notify(
    ctx context.Context,
    alert model.Alert,
    channels []model.NotificationChannel,
) error {
    for _, d := range m.dispatchers {
        if err := d.Notify(ctx, alert, channels); err != nil {
            return err
        }
    }
    return nil
}
```

## 五、规则与通知渠道的绑定

在规则定义中，`Channels` 字段直接内嵌了通知渠道配置：

```json
{
  "name": "HighMemoryUsage",
  "expr": "avg(container_memory_usage_bytes) / 1024 / 1024 > 1024",
  "severity": "warning",
  "interval": "60s",
  "channels": [
    {
      "type": "webhook",
      "config": {
        "url": "https://hooks.example.com/alert"
      }
    },
    {
      "type": "dingtalk",
      "config": {
        "webhook_url": "https://oapi.dingtalk.com/robot/send?access_token=xxx",
        "at_mobiles": "13800138000"
      }
    }
  ]
}
```

这种设计让告警通知配置与告警规则同生命周期——创建规则时一并配置通知渠道，删除规则时一并删除，无需额外管理。

## 六、分发模块的可扩展性

当前的分发模块具备良好的扩展性，新增渠道只需：

1. 在 `NotificationChannel.Type` 中注册新类型
2. 在 `ChannelDispatcher.Notify` 的 switch 中添加新 case
3. 实现对应的发送方法（如 `sendSlack`、`sendWechat`）

无需修改其他模块的代码。

## 七、当前设计局限

1. **缺少异步发送**：当前发送是同步的，邮件/SMTP 可能耗时较长，影响评估链路。未来应引入消息队列或 goroutine 池异步发送。
2. **缺少发送成功/失败的状态记录**：当前无法查询某条告警是否成功送达。
3. **邮件密码明文**：配置文件中明文存储 SMTP 密码，存在安全风险。
4. **重试策略简单**：固定间隔重试，未实现指数退避或最大重试时间窗口。
5. **缺少通知模板**：当前消息内容是代码中硬编码的，无法自定义模板。

## 总结

告警分发模块的设计要点：

- **接口抽象**：统一的 `Dispatcher` 接口，支持多实现
- **配置分层**：全局默认配置 + 规则级覆盖配置
- **多渠道支持**：Webhook（通用）、邮件（传统）、钉钉（企业IM）
- **签名安全**：钉钉 HMAC 签名支持
- **可扩展架构**：新增渠道只需添加新 case，不影响现有代码

通过这套分发机制，alert-engine 能够将告警信息及时送达值班人员手中，完成告警系统的闭环。
